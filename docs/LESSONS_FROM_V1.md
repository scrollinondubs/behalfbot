# 30 Lessons from Building an Autonomous Claude Code Assistant

By Sean Tierney (Vibecode Lisboa). Bundled with the *Replace OpenClaw with Claude Code* talk recording and slides.

> **Source-of-truth note (2026-05-05):** This document is captured verbatim from Sean's 2026-05-05 Substack post + Google Doc. Treat it as the canonical install-history record. Subsequent chassis design references lessons by number — do not renumber. New lessons get appended (#31, #32, ...) as we hit them during installer rollouts.

---

After 30 days trying to make OpenClaw work as an autonomous AI assistant, I switched to Claude Code and built something that actually does the job. Two weeks in, here are the 30 lessons that would have saved me a lot of pain — and that apply to anyone trying to set up a persistent AI agent.

1. **The platform features you think you need, you don't.** Going in I assumed Dispatch, /loop, and /voice would be load-bearing — that I'd be wiring tasks through Anthropic's official primitives. None of them mattered. Discord channels handle remote tasking, a custom heartbeat dispatcher beats /loop for scheduled work, and Discord voice notes plus local Whisper replaced /voice cleanly. The actual building blocks are unglamorous: channels for inbound, MCPs and CLIs for tool access, and computer-use as the fallback for everything else.

2. **tmux is the seatbelt of every headless agent setup.** Without it, closing your laptop kills the SSH session and takes Claude Code down with it. The mental model that worked: closing your laptop without tmux is unplugging the power cord, with tmux it's only unplugging the monitor cord — the process keeps running, you just stop watching. Bonus: SSH-started shells have a different security context than GUI terminals (no macOS keychain integration), which surfaces in OAuth token refresh, another reason to start every long-running process inside tmux.

3. **OAuth tokens expire ~8 hours headless and you don't fight that, you build around it — but the V1 keepalive pattern is now harmful.** Claude Code's auth token has an ~8h TTL, and headless SSH sessions have no macOS keychain to silently refresh it overnight. Original V1 answer: a layered seatbelt of a 6h keepalive launchd job (`claude --print "ping"`), a watchdog, and a 5:30 AM pre-emptive respawn. **Skip the keepalive.** See lesson 36 — Anthropic's OAuth refresh-tokens are single-use, so forcing a refresh on a schedule rotates the token server-side and invalidates the copy held in-memory by every other live `claude` process. The current canonical answer is the OAuth bridge sync (`chassis/scripts/sync-claude-oauth-bridge.sh`, every 30 min): it propagates lazily-refreshed tokens between macOS Keychain and `~/.claude/.credentials.json` bidirectionally, so all peer `claude` processes converge without any forced rotation. Keep the watchdog (still useful) and the 5:30 AM respawn (still useful for fresh process state, not for token freshness). Drop the 6h keepalive.

4. **Discord-channel mode is incompatible with auto-mode today, period.** Run Claude Code with the channel plugin and Anthropic's auto-mode classifier on at the same time, and every approval-gated tool call silently stalls — the inbound message arrives, the model wants to act, the approval surface tries to render to a terminal nobody's watching, and nothing happens. The current workaround is `--dangerously-skip-permissions` plus deterministic guardrails enforced at the hook layer (`.claude/hooks/guardrails.sh`). You enforce safety as code, not as runtime prompts. Eventually the two will coexist; today they don't.

5. **When official APIs hit a permission wall, Playwright against the logged-in browser is the answer.** Tried three Slack integration paths — bot tokens, user OAuth, direct API — all needed workspace-admin approval. LinkedIn was the same story. What worked: Playwright reading the web UI as the already-authenticated session, no new permissions beyond cookies that already exist. The "official" path is a dead end for read-mostly automation against platforms that gate API access behind enterprise tiers; embrace the headless browser, it's not a hack, it's the design.

6. **Treat the AI like an employee with its own accounts, graduated trust, and hard limits.** Don't reuse your iCloud, GitHub, or Workspace identity for the agent — give it a separate Apple ID, a separate GitHub handle, its own MCP credentials. Start with a tight allowlist (read everything, approve before writing, never spend money, never email externally) and expand the autonomy envelope as the system earns trust. The hard limits live in CLAUDE.md so they survive context resets, and they're enforced at the hook layer where the LLM can't talk its way around them.

7. **Gather first, prompt second — let cheap shell scripts decide whether the model gets to run.** The dispatcher fires every 15 minutes but `claude -p` only invokes when a lightweight gather command (jq, find, a Postgres select) returns work. Net: 96 dispatcher runs a day, ~4 actual model invocations, the rest cost effectively nothing. The pattern generalizes anywhere you'd be tempted to "just call the LLM and see if there's anything to do" — put a deterministic check in front of it and the bill drops by an order of magnitude.

8. **Issuing `/clear` over Discord kills the channel pairing — context resets aren't scoped to the conversation.** When you /clear from Discord, Claude Code's plugin detaches the channel binding alongside the conversation history. Inbound messages keep arriving but nothing inside Claude Code is listening anymore. The fix isn't avoiding /clear — sometimes you genuinely need a fresh context — it's a keepalive watchdog that re-establishes the channel pairing on the other side of any reset. Treat channel state as orthogonal to conversation state; they look related and they aren't.

9. **Pre-emptively respawn Claude Code on a schedule for fresh process state, not for token freshness.** The /login reauthentication timeout will eventually corner you — an 8h token plus an interactive re-login flow that nobody's there to satisfy. The pattern that worked: a launchd-triggered restart of the whole tmux session at 5:30 AM, before the morning briefing fires and before any inbound Discord traffic of the day. Failures shift from "the briefing didn't run because auth died at 4 AM" to "auth died gracefully at 5:29 and was healthy at 5:31." The watchdog handles unscheduled cases; the cron handles the predictable one. **Important update post-V1:** the respawn's value is now fresh process state (memory cleared, MCP reconnections, channel pairing re-established), NOT token freshness. Token freshness is the bridge sync's job (see lesson 3 + 36). Do NOT add a separate 6h token keepalive on top — it'll race the bridge and invalidate peer processes.

10. **For apps with no web interface, an Android emulator plus UI Automator XML dumps is the universal escape hatch.** Hinge has no public API and the iOS app is impossible to automate without a jailbroken device. The path that worked: a Pixel emulator on the Mac Mini, ADB for input, and `uiautomator dump` for reading the rendered view tree as XML. From there it's just selectors against a different DOM — same mental model as Playwright, different surface. For any closed mobile app, treat it like a web page rendered into a different tree, and the tooling exists to drive it.

11. **Scheduled features can be fully scaffolded and still dead if not registered in the dispatcher config.** Scripts, prompt, state entry all exist — but if there's no declaration in HEARTBEATS.md, the dispatcher never invokes them. Silent dormancy. Register ANY new heartbeat in the same PR that adds its scripts, or add it to a pre-deploy checklist.

12. **Auto-throttles need new-block auto-lift, not just high-usage auto-enable.** Our quota-monitor auto-enables conservation mode at 80% utilization, but only auto-disables on a very narrow window (util < 20% AND >240min remaining). Once tripped, the state persists across sessions because almost no new session satisfies that disable condition. Lesson: for every auto-on switch, the auto-off needs to fire *more* aggressively than you think — disable on any new billing block detected, regardless of current util.

13. **Sharing destructive-read state between two heartbeats is a race waiting to happen.** Telegram Bot API's `getUpdates` consumes each offset on read. Both telegram-groups heartbeat and morning-briefing called the same gather script sharing one state file — whoever fired first won, the other reported "zero messages." Fix: the cheap caller reads a cached digest file, doesn't re-query the destructive source.

14. **Merge-conflict markers in a JSON config silently break every consumer.** `git stash pop` after multi-file edits leaves `<<<<<<< Updated upstream` markers. `jq .enabled` on that file errors out, your script's `|| echo false` fallback treats it as "disabled," and the system silently reverts behavior. Lint config files in CI or at minimum `jq empty` them pre-commit.

15. **`gh issue list --label "a,b"` requires BOTH labels (AND semantics), not "either" (OR).** Most devs read the comma as "or." A picker with `--label "ready,<v1-reference-install>"` never matches any real issue. Use `gh search issues --label ready --no-assignee` instead if you want a single-label match across all repos.

16. **UserPromptSubmit hooks don't fire on Discord-channel inbound messages.** They fire on directly-typed prompts only. If your proof-of-life hook expects to capture channel messages, it won't — the file goes stale even while you're actively messaging. Query the channels directly from the gather script; don't rely on hooks catching a broader surface than they actually do.

17. **Vercel Deployment Protection silently breaks automated Playwright testing against preview URLs.** The SSO redirect returns 200 and a "login" page loads — tests technically "pass" while never hitting the app. Need a Protection Bypass for Automation token + `?x-vercel-protection-bypass=<token>&x-vercel-set-bypass-cookie=true` query params. Takes 2 min to set up once you know it exists.

18. **Google Workspace primary-domain flip does NOT auto-rename users.** You still have to manually rename each account (e.g. `sean@charitymakeover.org` → `sean@vibecodelisboa.com`). Every "Sign in with Google" OAuth session keeps working because the binding is the Google `sub` (stable user ID), not the email. So renaming is safer than it feels — Atlassian / Loom / Stripe don't care — but you still do it one deliberate click at a time.

19. **DKIM record values get chunked by DNS hosts into 255-char quoted strings.** Most docs show the whole key as one line. The receiver reassembles automatically. Don't panic when your Namecheap DNS panel shows a 400-char key split into two quoted segments — that's correct and `dig` will confirm it resolves.

20. **The dispatcher gather script is the right place for expensive checks to no-op cheaply.** Every 15 min is fine IF the gather short-circuits on trivial conditions (feature disabled, out-of-window, conservation mode) before any paid API call. Net: 96 runs/day for ~$0 in tokens, only ~4 actually fire `claude -p`. Pay-to-play only on the work itself.

21. **Adding "negative references" to a similarity-based scorer can collapse scores to zero.** Our CLIP dating-photo scorer regressed to all-zeros the moment we started auto-populating negative reference images from "pass" decisions. Root cause: `img_score = img_top3_positive - img_top3_negative`. A legitimate face matches both positive and negative references roughly equally, the subtraction cancels out, every raw score falls below the rescaling floor, clamped to 0. Fix: weight negatives at 0.3× positives AND curate strictly (only aesthetic-dissimilar passes, not age or vibe passes).

22. **n8n's `splitInBatches` iterates over INPUT items, not nested array fields.** If your upstream HTTP node returns one response body with a `.logs` array inside, `splitInBatches` loops once over the outer response, never the inner array. Zero downstream work, no error, no warning. Add an explicit "fan out" Code node between fetch and loop that does `return logs.map(x => ({json: x}))`.

23. **Webhooks that send email need to persist the returned Gmail `threadId` immediately after send.** Otherwise your reply-tracking workflow can never correlate bounces to the original send, and `users.email_bounced_at` + `email_outreach_log.bounced_at` diverge. ${ASSISTANT_NAME} Email Sender had this gap for weeks — every LBS cohort outreach was invisible to bounce detection. Fix is one PATCH call right after `drafts/send`.

24. **When testing an auto-picker, the trigger conditions matter more than the query logic.** "Why doesn't it fire?" usually means a gate is silently failing (conservation mode, util threshold, time window, dispatcher not seeing the heartbeat at all). Debug each gate independently before touching the query. A working query that never runs is indistinguishable from a broken query.

25. **Three failure patterns recur:**
    - **Silent dormancy** — feature scaffolded but a single piece of config never flipped (lessons 11, 16, 22, 24).
    - **Subtraction or race collapse** — two mostly-correct pieces of state math combined into a degenerate zero state (lessons 13, 14, 21).
    - **Implicit assumptions about API behavior** — "surely it does X" where the docs or reality said Y (lessons 12, 15, 17, 18, 19, 23).

26. **LaunchAgents pause without a logged-in user; LaunchDaemons survive reboots.** The Mac Mini auto-rebooted at 06:08 to install a macOS update. Every ${ASSISTANT_NAME} service in `~/Library/LaunchAgents/` then sat dormant for the next two and a half hours because they're all bound to the Aqua GUI session — `LimitLoadToSessionType=Aqua` is on by default for user-installed agents. No one was logged in, so the heartbeat dispatcher, the Discord watchdog, the Postgres bootstrap, and the FastAPI dashboard all stayed loaded but inactive. The morning briefing fired late, <v1-reference-install>-discord stayed dead until manually restarted, the briefing URL returned 502 because the dashboard wasn't running. None of those services were broken — they were waiting for a user to log in. Fix is to move anything reboot-critical into LaunchDaemons in `/Library/LaunchDaemons/` with the `UserName=<v1-reference-install>` key. Daemons run at boot before any login, drop privileges to the named user, and survive everything short of a kernel panic. LaunchAgents are fine for user-session things (anything that needs the keychain, anything Sean-facing with GUI). Infrastructure goes in daemons.

27. **Substring-grep guardrails false-positive on documentation inside heredocs.** The shell guardrail that blocks mutating curl/wget calls to non-allowlisted hosts was a naïve `grep -E '(curl|wget) ... -d'` over the entire bash command string. Worked fine for two months. Then I tried to file a `gh issue create` whose body documented `curl -X POST ... -d <hostname>` as an example, and the grep happily matched the text inside the heredoc body. The guardrail blocked the issue creation as if curl was being invoked. Two fixes layered together: anchor curl/wget detection to actual command boundaries (start-of-string, after `;`, `&`, `|`, `(`, backtick, `$()`) so text inside string literals stops matching, and explicitly early-skip trusted CLI wrappers like `gh` and `tailscale` whose body content is documentation, not execution. General lesson: when a guardrail's job is to reason about what a shell command does, regex over the raw command string is brittle. Either parse the command properly, or scope the check to a known-safe wrapper list.

28. **What matters for behavior is what's on disk, not in the merge log.** A merged PR is not the same as a deployed change when launchd reads scripts from disk. The ${ASSISTANT_NAME} heartbeat dispatcher reads scripts at fire time, not from any in-memory or git-aware cache. So a PR can land on `origin/main` cleanly — review approved, CI green, GH UI shows merged — and the production behaviour does not change at all if the local checkout was left on a stale feature branch and `git pull` never ran. Hit this pattern twice in two weeks. The first time we treated as a one-off. The second time made it clear that vigilance is not the fix; detection is. Shipped a repo-drift heartbeat that runs every 30 min, fetches `origin/main` read-only across all tracked repos, and pings `#<ops>` if any commits behind touch heartbeat-runtime paths (`scripts/`, `scheduled-tasks/`, `skills/`, `.claude/hooks/`, `services/`). Detection-only — no auto-pull, because the right remediation depends on whether the local checkout has WIP and only the human knows that. The general lesson: when production state is "whatever's on disk right now," a merged PR is a story about the past; what matters is whether the disk has been told.

29. **Single-tool reverse-image search false-positives in noisy domains — triangulate before you act.** First catfish detector ran Yandex similarity on profile photos, counted "different names attached on aggregator sites" as identity-theft evidence, and recommended cancellation. It worked correctly on two real catfish (photos lifted from public OnlyFans-leak aggregators) and then false-positived on a real Russian-Lisbon expat whose only crime was matching the generic "blonde 30s" aesthetic of every blonde 30s Yandex indexes. Fix: never act on a single tool's verdict. Require consensus across **TinEye** (exact-pixel), **Google Lens** (face + celebrity ID), and **PimEyes** (face geometry); all three agreeing on theft = catfish, one or two disagreeing = real person, walk the cancellation back. Heuristics that work on the head of the distribution often collapse on the tail; the cost of a false-positive cancellation is high enough that consensus isn't optional.

30. **"The agent can read everything" doesn't scale ethically — privacy boundaries are surface-specific.** Started with full read access to WhatsApp, Twitter, LinkedIn, and Slack on the assumption that "I already have access, so the agent does too." Then realized: WhatsApp DMs carry a privacy expectation from the *other party* that I can't unilaterally consent to share, while group messages don't because group context is already disclosed to N participants. Fix is per-surface: WhatsApp narrowed to a wrapper script + group allowlist + hook-layer block on raw `wacli messages`; Twitter, LinkedIn, and Slack queue different designs because they use Playwright not a CLI. The general principle is default-deny per surface, allowlist explicitly, and document the gaps — privacy expectations differ by surface (DM vs group, public profile vs feed) and the right model isn't all-or-nothing.

31. **`uv` is the chassis-canonical Python version manager — skip the deadsnakes/pyenv detour.** Ben's Debian 12 box shipped Python 3.11.2 (system default). Chassis requires 3.12+ (f-string backslash handling, PEP 701). First instinct was to apt-add deadsnakes PPA or build from source. Wrong path. The one-liner `curl -LsSf https://astral.sh/uv/install.sh | sh && uv python install 3.12` installs user-locally in ~55s without touching system Python. `uv run --python 3.12 python script.py` gives you 3.12 immediately; `python3` still resolves to the system version for non-chassis use. The `uv run --with <pkg>` ephemeral-dep pattern also eliminates most venv proliferation inside chassis scripts. Ship this as the Step 1 bootstrap path, not a fallback.

32. **When a private repo has no installer bot-account yet, tarball-ship the chassis.** `git clone https://github.com/scrollinondubs/behalfbot.git` requires authentication. Ben hadn't provisioned his bot GitHub account yet. Rather than block, export with `git archive --format=tar.gz --prefix=behalfbot/ origin/main` from the operator's local clone, `scp` to the installer's box, untar to `~/behalfbot/`. Loses `.git` history but ships the exact working tree bootstrap reads. Once the bot account exists, re-init non-destructively: `cd ~/behalfbot && git init && git remote add origin git@github.com:scrollinondubs/behalfbot.git && git fetch origin main:main && git switch main`. Skip `--hard` resets — the fetch+switch path is non-destructive.

33. **"Plugin" and "core module" are different concepts — don't conflate them in chassis.config.yaml.** installer-1's config had `admin: true`, `briefing: true`, `crm: true`, `outreach.enabled: true` — four modules with no `plugins/<name>/` directory. They're implemented in `chassis/scripts/` (briefing-server.py etc.) as chassis-core capabilities, not opt-in plugins. The validate script initially refused them as "missing plugin dirs." Fix: differentiate core modules (ship with chassis, always-available) from opt-in plugins (expect `plugins/<name>/` dir). V1 stopgap: skip dir-check for known core module names. V2 recommendation: rename the config schema to `modules.core.*` vs `modules.plugins.*`, or extract admin/briefing/crm/outreach to `plugins/<name>/` for full symmetry. Until that lands, any new validator must read the core-module list explicitly.

34. **Start on Postgres, not SQLite — the migration cost is real.** chassis.config.yaml shipped with `database: sqlite` as the V1 default with a comment "Postgres = scale-up path, not V1." installer-1's install flipped this on Day 1 on Sean's directive "remember how involved that migration was after the fact." The SQLite→Postgres migration on the V1 reference took several hours of schema translation, data re-import, and DSN plumbing. A fresh installer doing it up front takes 10 minutes (`postgresql-15` via apt, `createdb behalfbot`, peer auth, four env vars). The "scale-up path" framing is wrong — Postgres-from-start is just correct. Ship `database: postgres` as the default in chassis.config.yaml and update the comment.

35. **systemd unit files for the dispatcher must embed `PATH` explicitly — uv and uv-installed binaries won't resolve otherwise.** Ben's heartbeat dispatcher fired but `uv run` inside heartbeat scripts failed silently because the systemd-user unit inherited a stripped-down PATH. Fix: embed `Environment=PATH=%h/.local/bin:/usr/local/bin:/usr/bin:/bin` in the `.service` file. Without this, every heartbeat that invokes `uv run` (or any Homebrew / user-local binary) silently no-ops. This is distinct from the macOS LaunchDaemon `EnvironmentVariables` key — both need the same fix, but the syntax differs. Ship the PATH line in both the systemd and launchd templates under `chassis/scheduled-tasks/`.

36. **Every assumption in INSTALL_PROFILE.md will be wrong until proven by install-time probe.** The pre-install profile for Ben was drafted from interview material and ratified — then every field that described Ben's existing setup got corrected at install time: VW port (8443 → 8222), VW master account email (assumed ozzy@ → actual ben@benlakoff.com), Strava/FDC opt-in assumptions, Notion scope (full workspace → narrow LP+Startup+1 page). Pattern: interview material reflects intent; live system reflects reality. The Confabulator interview can close this gap for intent-fields (preferences, desired scope). It cannot close it for existing-system fields. Chassis hydration should always probe existing systems (curl VW URL, `whoami`, `ollama --version`) and surface mismatches before any irreversible step — "what you told us" vs "what we found."

37. **Claude Code session-state-loss between separate conversations is an install blocker — write install state to memory, not to session context.** Day 2 of installer-1's install opened with the assistant claiming "no SSH session has happened yet." The prior session had logged "Step 10 ✓ — systemd-user timer running on fatboy" in Discord. Session memory didn't carry forward. On multi-day installs the LLM will also forget that INSTALL_PROFILE.md and chassis.config.yaml were ratified the night before — the install then backtracks. Fix: every install-step transition writes a `set_entity` to the installer's memory file (equivalent to `project_lakoff_install_state.md`). At the top of every installer-driving session, read that file first. "What's been done" is a durable fact that must survive context resets; session memory isn't durable.

38. **Vaultwarden spin-up belongs in the Phase-2 SSH session, not in installer pre-work.** First installer-3 install (case study #3, started 2026-05-18) surfaced a recurring pattern: installers hit Step 1f of the homework doc, read "stand up Vaultwarden in Docker," and either (a) get blocked on Docker Desktop, (b) get blocked on docker-run command syntax, or (c) misread the step and try to run `setup-wizard` locally. Sean's directive 2026-05-18 settled it: since Sean+${ASSISTANT_NAME} are already SSH'd into the installer's box for the chassis bootstrap, just spin Vaultwarden up at the same time as part of the install transcript, validate it with a smoke curl, hand the master credentials to the installer at signoff. The installer then adds their agent-account credentials (Google bot, GitHub bot, Discord bot, plugin keys) into the running vault themselves. The pre-existing fallback "use existing Bitwarden cloud account" stays valid for installers who already use Bitwarden. Net: Phase-1 work for the installer shrinks to browser-only account provisioning, no Docker, no terminal, no Vaultwarden setup. Reduces cognitive load AND fixes the doc-confusion pattern where Step 1f reads as Phase-1 work but is actually Phase-2 work. Action items: amend `docs/installer-homework.md` Step 1f to read "Vaultwarden handled at install time — skip this step (existing Bitwarden cloud account is also valid)"; amend `chassis.config.yaml` template `identity_isolation.secrets_handoff` to default `vaultwarden_phase2_or_existing_bitwarden` rather than the current `vaultwarden`.

40. **Heartbeats that surface candidates must propose, not auto-file — even a high-quality gate produces unapproved work.** V1 reference install's pulse-triage heartbeat evaluated daily Claude Code Pulse + Agent Builders Telegram URLs against a beneficial+feasible+dedup gate, then called `gh issue create` directly when both passed. The gate worked — only 2-3 items filed per day, no spam — but Sean noticed on 2026-05-31 that 5 unapproved issues had accumulated across 3 days. The structural problem: the issue tracker is a contract about future attention, and auto-filing violates that contract regardless of how good the gate is. Fix (`scrollinondubs/new-jaxity#127`, merged 2026-05-31): every YES/YES item now posts a structured ratification proposal to the operator's channel (case / effort / risk / my take + emoji-react instructions). Filing only happens after the operator's 👍. The heartbeat's state file gains a `pending_proposals[]` list; the `issues_filed_today` counter is renamed `proposals_posted_today`. **Generalisation for chassis canonical:** any heartbeat that surfaces "should we do X?" must propose-first, file-on-ratify. The propose-first rule is now documented as anti-pattern #23. Apply the same pattern to any future surface that wants to inject work into the operator's queue.

39. **Forced OAuth refresh-token rotation invalidates every peer `claude` process holding the old token in memory.** V1 reference install ran `com.<assistant>.claude-auth-refresh` for months — a launchd job firing `claude --print "ping"` every 6h, originally added (per lesson 3 of the V1 lessons) to keep the access-token fresh during idle overnight windows. Anthropic's OAuth refresh-tokens are single-use; the `ping` triggered a refresh, which rotated the refresh-token server-side. Every other live `claude` process (Discord intake `claude --channels`, interactive terminal `claude`, in-container dispatcher's `claude -p` subprocesses) still held the OLD refresh-token in memory. Their next refresh attempt → `invalid_grant` → "Not logged in." Observed pattern: 3x-daily auth invalidation on the Discord session, matching the keepalive's fire schedule (03:46 / 09:46 / 15:46 / 21:46 local). Diagnosis took weeks because the keepalive was the *fix* for a real V1 problem — until the OAuth bridge sync (chassis#138, `chassis/scripts/sync-claude-oauth-bridge.sh`) landed in May 2026 and made the keepalive obsolete + harmful. **Lesson for chassis canonical:** never auto-create or document a forced-rotation keepalive in a deployment with multiple concurrent `claude` processes. The bridge sync handles freshness via lazy refresh-on-use + bidirectional file↔keychain propagation, with no forced rotation. If you inherit a customer install that still has one, unload + delete it — see V1-reference removal in new-jaxity#89 for the template.
